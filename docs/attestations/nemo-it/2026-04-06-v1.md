## Findings

### CRITICAL

#### CRIT-001: Next.js Vulnerable Version (4 CVEs)

- **Category:** Dependency
- **Where:** `frontend/package.json` -- `next: ^14.2.5` (installed: 14.2.35)
- **CVEs:**
  - GHSA-9g9p-9gw9-jx7f -- DoS via Image Optimizer remotePatterns
  - GHSA-h25m-26qc-wcjf -- HTTP request deserialization DoS with insecure RSC
  - GHSA-ggv3-7p47-pfv8 -- HTTP request smuggling in rewrites
  - GHSA-3x4c-7xq6-9pq8 -- Unbounded next/image disk cache growth
- **Business Risk:** An attacker could crash the frontend, smuggle malicious requests past security controls, or exhaust server disk space -- causing service outages.
- **Remediation:** PROGRAMMATIC_FIX
- **Fix:** Upgrade Next.js to >= 15.5.14 or 16.2.2. Run `npm install next@latest` and verify build passes.

### HIGH

#### HIGH-001: Missing HTTP Security Headers (Frontend)

- **Category:** Configuration
- **Where:** Frontend response headers (http://localhost:3000)
- **OWASP:** OTG-CONFIG-002, OTG-CONFIG-003
- **Missing Headers:**
  - `X-Frame-Options` -- allows clickjacking attacks
  - `X-Content-Type-Options` -- allows MIME-sniffing attacks
  - `Content-Security-Policy` -- no XSS mitigation
  - `Strict-Transport-Security` -- no HTTPS enforcement
  - `Referrer-Policy` -- leaks referrer information
  - `Permissions-Policy` -- no feature restrictions
- **Business Risk:** Without these headers, the admin panel is vulnerable to clickjacking (an attacker embeds your admin UI in an iframe and tricks users into clicking things), XSS amplification, and MIME confusion attacks.
- **Remediation:** PROGRAMMATIC_FIX
- **Fix:** Add `headers()` configuration to `next.config.mjs` or add a custom `middleware.ts` that sets security headers on all responses.

#### HIGH-002: Missing HTTP Security Headers (Admin Panel)

- **Category:** Configuration
- **Where:** Admin Panel response headers (http://localhost:8880)
- **OWASP:** OTG-CONFIG-002, OTG-CONFIG-003
- **Missing Headers:** Same set as HIGH-001. Also exposes `server: uvicorn` header which reveals technology stack.
- **Business Risk:** Same clickjacking/XSS risk as HIGH-001, plus the server header tells attackers exactly what framework to target.
- **Remediation:** PROGRAMMATIC_FIX
- **Fix:** Add FastAPI middleware to set security headers and suppress the `server` header.

#### HIGH-003: AI Safety Module Not Implemented

- **Category:** AI Safety
- **Where:** `admin-panel/app/ai/safety/` -- directory does not exist
- **OWASP:** N/A (AI-specific)
- **Details:** The base AI provider (`admin-panel/app/ai/providers/base.py`) imports from `app.ai.safety` but gracefully degrades when the module is missing. This means ALL AI safety controls (prompt sanitization, PII masking, output validation, error sanitization) are currently **disabled**. The code paths exist but are no-ops.
- **Business Risk:** Without prompt sanitization, an attacker who can influence AI inputs (e.g., through API request data that gets analyzed) could inject malicious prompts. Without PII masking, sensitive data in API requests could be sent to external AI providers. Without output validation, the AI could return harmful content.
- **Remediation:** TECHNOLOGICAL_CONTROL
- **Fix:** Implement the `app.ai.safety` module with the four expected sub-modules: `sanitize.py`, `validate.py`, `pii_masker.py`, `errors.py`.

### MEDIUM

#### MED-001: Dockerfile Missing USER Directive (Development Stage)

- **Category:** Infrastructure
- **Where:** `frontend/Dockerfile` -- development stage (line 12-18)
- **OWASP:** OTG-CONFIG-005
- **Details:** The development stage of the frontend Dockerfile runs as root. The production stage correctly uses `USER nextjs` (line 45), so this only affects local development containers.
- **Business Risk:** Low in practice -- this is a development-only container. But if compromised, the attacker has root access inside the container.
- **Remediation:** PROGRAMMATIC_FIX
- **Fix:** Add `USER node` to the development stage after the COPY instruction.

#### MED-002: Kong Admin API Exposed on Host Network

- **Category:** Configuration
- **Where:** `docker-compose.yml` line 194 -- port 8001:8001
- **OWASP:** OTG-CONFIG-002
- **Details:** The Kong Admin API is mapped to the host on port 8001 (remapped to 8801). This API provides full control over Kong routing, plugins, and consumers -- unauthenticated. The docker-compose file notes "local dev only - never expose in prod" but the binding is to all interfaces.
- **Business Risk:** Anyone on the same network can view/modify all API gateway routes, add malicious plugins, or disable security controls. This is expected for local dev but must never reach production.
- **Remediation:** PROCESS_CONTROL
- **Fix:** For local dev, bind to 127.0.0.1 only: `"127.0.0.1:${KONG_ADMIN_PORT:-8001}:8001"`. Verify k8s deployments do NOT expose the admin API (confirmed: k8s network policies exist).

#### MED-003: Default Credentials in Docker Compose

- **Category:** Configuration
- **Where:** `docker-compose.yml` lines 32, 36, 39, 64, 230-232
- **Details:** Default passwords are set via env var fallbacks: `postgres_local_dev`, `kong_local_dev`, `admin_local_dev`, `redis_local_dev`, `local-dev-secret-key-change-in-production`, `mock-client-secret`. These are clearly labeled as dev defaults and use `${VAR:-default}` syntax, meaning they are overridden when `.env` is present.
- **Business Risk:** Low for local dev. The risk is that these defaults could accidentally be used in staging/production if `.env` is not properly configured.
- **Remediation:** PROCESS_CONTROL
- **Fix:** Add a startup check script that validates all secrets are overridden from defaults before the app starts in non-development environments.

#### MED-004: Grafana Default Admin Credentials

- **Category:** Configuration
- **Where:** `docker-compose.yml` line 389-390
- **Details:** Grafana admin is `admin/admin` by default. While env var overridable, the defaults are active for local dev.
- **Business Risk:** Same as MED-003. If Grafana is exposed, anyone can log in as admin.
- **Remediation:** PROCESS_CONTROL
- **Fix:** Override via `.env` in all non-local environments.

### LOW

#### LOW-001: Docker Compose Frontend Missing HEALTHCHECK in Dockerfile

- **Category:** Infrastructure
- **Where:** `frontend/Dockerfile` (detected by Trivy DS026)
- **Details:** The root-level Dockerfile has no HEALTHCHECK instruction. However, docker-compose.yml defines healthchecks for all services, so this is effectively mitigated for orchestrated deployments.
- **Remediation:** PROGRAMMATIC_FIX
- **Fix:** Add `HEALTHCHECK CMD wget --quiet --tries=1 --spider http://localhost:3000 || exit 1` to the production stage.

#### LOW-002: Server Header Disclosure (uvicorn)

- **Category:** Information Disclosure
- **Where:** Admin Panel HTTP responses
- **OWASP:** OTG-INFO-002
- **Details:** The `server: uvicorn` header reveals the backend technology.
- **Remediation:** PROGRAMMATIC_FIX
- **Fix:** Add `--header server:""` to uvicorn config or set via middleware.

#### LOW-003: AI Failover Provider Hardcoded API Key

- **Category:** Code Pattern
- **Where:** `admin-panel/app/ai/providers/failover.py:32`
- **Details:** The failover provider constructor passes `api_key="failover"` as a placeholder. This is not a real credential -- it's a sentinel value for the multi-model failover wrapper that delegates to the primary/fallback providers for actual API calls.
- **Business Risk:** None. This is an architectural pattern, not a leaked secret.
- **Remediation:** ACCEPT_RISK

## Recommendations (Priority Order)

1. **CRITICAL -- Upgrade Next.js** (CRIT-001): `npm install next@latest` in frontend/. This resolves 4 CVEs including request smuggling and DoS.

2. **HIGH -- Add security headers** (HIGH-001, HIGH-002): Add `X-Frame-Options`, `Content-Security-Policy`, `X-Content-Type-Options`, `Strict-Transport-Security`, `Referrer-Policy`, `Permissions-Policy` to both frontend and admin panel responses.

3. **HIGH -- Implement AI safety module** (HIGH-003): Create `admin-panel/app/ai/safety/` with `sanitize.py`, `validate.py`, `pii_masker.py`, `errors.py`. This activates the existing safety hooks in the provider base class.

4. **MEDIUM -- Bind Kong admin to localhost** (MED-002): Change docker-compose port binding to `127.0.0.1:8001:8001`.

5. **MEDIUM -- Add non-root user to dev Dockerfile** (MED-001): Add `USER node` to the development stage.

6. **LOW -- Add HEALTHCHECK to Dockerfile** (LOW-001): Add health check to production stage.

7. **LOW -- Suppress server header** (LOW-002): Remove `server: uvicorn` header.
